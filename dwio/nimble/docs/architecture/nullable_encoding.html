Stage 1 — Metadata (small reads from end of file)
Postscript (last 20 bytes) → footer size, checksum type
Footer (272 bytes) → stripe count, row count, optional section locations
StripeGroup (5,480 bytes) → per-stream offset + size within each stripe
Optional sections → schema, vectorized stats, etc.
Gives you file_info, schema, stripes, streams tables — all without touching stripe data.
Stage 2 — Stream data (reads actual stripe bytes)
For the encodings table, there's no shortcut.
The encoding tree is embedded in the raw stream bytes — it's not described in any metadata section.
The tool uses stripe group offsets to seek to each stream, reads its bytes, then runs traverseEncodings to parse the tree from the headers.
Key insight: Each stream in a Nimble file is just an opaque blob of bytes. The structure is self-describing — every encoding type has a known binary layout baked into the code. There is no separate schema or tree descriptor.

The parser (traverseEncodings) and the reader (NullableEncoding constructor) use the exact same byte layout knowledge. There is no separate schema or tree descriptor — the binary format is the schema. Each encoding reads its header, carves out child byte ranges, and recursively constructs child encodings via EncodingFactory::create().

How is nulls_ stored? Two encoding choices

Option A: TrivialEncoding<bool> 7 bytes

Stores 1 bit per row. 50 rows = 50 bits = 7 bytes.

49 zero-bits stored just to record "not null". All wasted.

Option B: SparseBoolEncoding ~5 bytes

Stores only positions of the minority value. 1 non-null → store [37].

sparseValue_ tells you what the stored indices mean:
  • sparseValue_ = true → indices point to non-null (true) rows. Used when non-nulls are rare.
  • sparseValue_ = false → indices point to null (false) rows. Used when nulls are rare.

When does SparseBool win? When the minority fraction < 1/lg(n).
For n=2.9B: threshold ≈ 1/lg(2.9B) ≈ 1/31 ≈ 3.2%.
Your case: 258K / 2.9B = 0.009% — well below threshold.

When does Trivial win? When null rate is close to 50%. Both true and false positions are equally frequent, so storing indices costs more than storing all bits.
